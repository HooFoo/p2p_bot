# Техническое задание: Телеграм-бот для P2P обмена валют

## 1. Введение
Данный документ описывает требования к разработке Телеграм-бота для прямого (P2P) обмена валют между пользователями. Бот выступает в роли площадки для размещения заявок на обмен и поиска подходящих контрагентов.

## 2. Стек технологий
- **Backend:** PHP (версия 8.1+)
- **База данных:** MySQL (MariaDB)
- **API:** Telegram Bot API
- **Библиотеки:** Рекомендуется использование `nutgram/nutgram` или `irazasyed/telegram-bot-sdk` для работы с API Telegram.

## 3. Структура данных (База данных)

### 3.1. Таблица `users`
Хранение информации о пользователях бота.
- `id` (INT, PK) — Внутренний ID.
- `telegram_id` (BIGINT, UNIQUE) — ID пользователя в Telegram.
- `username` (VARCHAR) — Никнейм в Telegram.
- `first_name` (VARCHAR) — Имя пользователя.
- `rating` (DECIMAL, default 0) — Рейтинг пользователя (на будущее).
- `created_at` (TIMESTAMP)

### 3.2. Таблица `orders` (Заявки)
Хранение активных и закрытых заявок на обмен.
- `id` (INT, PK) — ID заявки.
- `user_id` (INT, FK) — ID создателя заявки.
- `sell_currency` (VARCHAR(3)) — Валюта, которую пользователь отдает (например, USD).
- `amount` (DECIMAL(15,2)) — Объем валюты.
- `buy_currencies` (JSON) — Массив валют, которые пользователь хочет получить (например, ["RUB", "EUR"]).
- `status` (ENUM: 'open', 'closed', 'partially_filled') — Статус заявки.
- `is_active` (BOOLEAN, default TRUE) — Активна ли заявка для поиска.
- `created_at` (TIMESTAMP)
- `updated_at` (TIMESTAMP)

### 3.3. Таблица `matches` (Сделки/Связи)
Хранение информации об откликах и контактах.
- `id` (INT, PK)
- `order_id` (INT, FK) — К какой заявке откликнулись.
- `responder_id` (INT, FK) — Кто откликнулся.
- `status` (ENUM: 'pending', 'accepted', 'rejected', 'failed') — Статус связи.
- `created_at` (TIMESTAMP)

## 4. Пользовательский интерфейс (Главное меню)
После команды `/start` пользователю доступно главное меню с кнопками:
1. **Мои заявки** — Список активных заявок пользователя.
2. **Хочу обменять** — Создание новой заявки или переход к поиску.
3. **История** — Список закрытых (завершенных) заявок.

## 5. Основные сценарии

### 5.1. Создание заявки ("Хочу обменять")
Процесс реализован в виде пошагового опросника (Flow):
1. **Выбор своей валюты:** Кнопки с популярными кодами (USD, EUR, RUB, и т.д.).
2. **Ввод объема:** Текстовое сообщение.
   - *Валидация:* Должно быть числом больше нуля.
3. **Выбор желаемых валют:** Множественный выбор (кнопки с галочками или переключением состояния). Минимум одна валюта должна быть выбрана. Кнопка "Готово".
4. **Завершение:** После сбора данных создается запись в `orders`.

### 5.2. Поиск и отклик
Сразу после создания заявки (или через отдельный поиск) бот ищет до 10 подходящих заявок от других пользователей.
- **Условие подбора:** `sell_currency` (другого юзера) входит в `buy_currencies` (текущего юзера), и наоборот. Объем должен быть сопоставим (или частичное перекрытие).
- **Отображение:** Список кнопок с Именем продавца.
- **Действие:** При клике на кнопку отправляется "Отклик" создателю выбранной заявки.

### 5.3. Подтверждение сделки и обмен контактами
1. Создатель заявки получает сообщение: "На вашу заявку [Описание] откликнулся [Имя]". Кнопка: "Принять".
2. При нажатии "Принять":
   - Статус связи в `matches` меняется на `accepted`.
   - Отправителю отклика присылается контакт создателя (ссылка на профиль или @username).
   - Создателю присылается контакт откликнувшегося.
   - Сделка считается закрытой.
3. После закрытия сделки создателю приходит сообщение с вопросом: "Что сделать с остатком по заявке?". Кнопки:
   - "Изменить объем" (если сделка была частичной).
   - "Закрыть заявку" (сделать неактивной).
   - "Оставить без изменений" (сделать отметку о неудачной сделке, если что-то сорвалось).

## 6. Пользовательские истории (User Stories)

1. **Как пользователь**, я хочу видеть свои активные заявки, чтобы иметь возможность их быстро редактировать или закрывать.
2. **Как пользователь**, я хочу удобно выбирать валюты через кнопки, чтобы не совершать опечаток при вводе.
3. **Как пользователь**, я хочу получать уведомления, когда кто-то заинтересовался моей заявкой, чтобы быстро выйти на связь.
4. **Как пользователь**, я хочу видеть историю своих успешных обменов, чтобы отслеживать свою активность.
5. **Как пользователь**, я хочу, чтобы бот автоматически предлагал мне подходящих партнеров сразу после создания заявки, чтобы сэкономить время на поиск.

## 7. Функциональные требования
- Обработка команды `/start` и инициализация пользователя.
- Механизм состояний (FSM) для пошагового создания заявки.
- Валидация числовых полей.
- Логика подбора (Matching) на стороне БД.
- Система уведомлений между пользователями.
